同步和互斥

同步和互斥是进程之间竞争和协作的解决方法（两种进程关系）。同步是直接制约关系，在异步环境下进行，可以用信号量实现同步。互斥是间接制约关系，满足空闲让进（我有空你才能用）、忙则等待（我没空你就得等）、有限等待（当然我不会让你一直等）、让权等待（等待的时候你得把处理器还给OS）。消费者和消费者是互斥关系，消费者和生产者是同步关系。

**临界区(critical section)**是指包含有共享数据的一段代码，这些代码可能被多个线程访问或修改。 临界区的存在就是为了保证当有一个线程在临界区内执行的时候，不能有其他任何 线程被允许在临界区执行。对临界资源的访问必须互斥的进行

单标志法
int turn = 0;
P0:{
    while(turn != 0);
    do critical section;
    turn=1;
}
P1:{
    while turn != 1;
    do critical section;
    turn = 0;
}
//强制交换次序访问，不交换不能访问 


双标志先检查法
boolean flag[2] = {false,false};
P0:{
    while flag[1];
    flag[0] = true;
    do critical section;
    flag[0] = false;
    remainder section;
}
P1:{
    while flag[0];//先查
    flag[1] = true;//后改
    do critical section;
    flag[1] = false;
    remainder section;
}
//先查后改，就会同时进入临界区（进入区交替执行）。都玩阴的，说的和做的不一样，就会同时进入。

双标志后检查法
boolean flag[2] = {false,false};
P0:{
    flag[0] = true;
    while flag[1];
    do critical section;
    flag[0] = false;
    remainder section;
}
P1:{
    flag[1] = true;
    while flag[0];
    do critical section;
    flag[1] = false;
    remainder section;
}
//先改后查，可能死等导致饥饿（进入区交替执行）太老实，就会饿


皮特森算法
int turn = 0;
boolean flag[2] = {false,false};
P0:{
    flag[0] = true;
    turn = 1;
    while flag[1] && turn == 1;
    do critical section;
    flag[0] = false;
    remainder section;
}
P1:{
    flag[1] = true;
    turn = 0;
    while flag[0] && turn == 0;
    do critical section;
    flag[1] = false;
    remainder section;
}
//利用flag解决临界资源的互斥问题，利用turn解决饥饿问题 改让查

Semaphore(PV)​

// 整型信号量
wait(S){
    while(S<=0);
    S = S - 1;

    S = S + 1;

//记录型信号量
typedef struct{
	int value;
	struct process *Q;//阻塞进程的排队链表
} semaphore;

void wait(semaphore S){
    S.value --;
    if(S.value <0){
        block(S.Q);//运行态进入阻塞态
        add this process to S.Q;
    }
}

void signal(semaphore S){
    S.value ++;
    if(S.value<=0){
        remve a process P from S.Q;
        wakeup(P);//唤醒阻塞队伍中的第一个进程
    }
}

死锁

  1. 参与死锁的进程至少有两个

  2. 每个参与死锁的进程均等待资源

  3. 参与死锁的进程中至少有两个进程占有资源

  4. 死锁进程是系统中当前进程集合的一个子集

死锁产生的原因：不可剥夺性系统资源不足；进程推进顺序不当；信号量使用不当

死锁产生的必要条件：互斥（笔和纸同时只能一个人使用）-不剥夺（法制社会不能抢）-请求与保持（等待时间不会释放资源）-环路等待（你等我的笔，我等你的纸）

死锁预防​

会对系统并发性产生很大的副作用

破坏4个必要条件之一：

  1. 破坏互斥条件：不太可能，资源本身的属性限制

  2. 破坏不剥夺条件：获得了某些资源的进程，若新的资源请求不能立即满足，必须释放所有已有的资源。造成之前一段工作失效，重复申请和释放资源增加系统开销，降低系统吞吐量。通常不用于代价较大的场合

  3. 破坏请求保持条件：预先静态分配方法，一次性申请所有资源，没有满足前不投入运行。简单安全，但是降低了资源利用率。导致其它进程饥饿

  4. 破坏环路条件：有序资源分配方法，资源编号。确定每个进程必须按编号递增的顺序请求资源。同类资源一次申请完。

死锁避免：银行家算法​

dijkstra算法

将系统的状态分为安全状态和不安全状态，系统进行资源分配前先计算资源分配的安全性，若此次分配不会导致系统进入不安全状态，则分配，否则等待。若某一刻，系统能按某种顺序为每个进程分配所需的资源，直至最大需求，使每个进程都可以顺利完成，则此时的系统状态为安全状态，该序列为安全序列。不存在一个该序列，则此时的系统状态为不安全状态。
